    Appliance install (no host setup)¶

    This guide covers the appliance profile — the zero-host-setup path that works on any Linux machine with KVM. It runs the agent in a throwaway qemu/KVM VM with networking and an egress allowlist enforced inside the guest, so it needs no root, no bridge, no host firewall, no containers.

    Other profiles

    The hardened and GPU profiles (host-enforced egress via nftables + rootless podman) need more host setup and are Linux/Arch-leaning — see Hardened profile setup. This page is the appliance.

    Tested on Ubuntu/Debian; package names for Fedora and Arch are noted inline.

    Requirements¶

    • A Linux host with hardware virtualization (Intel VT-x / AMD-V) enabled in the BIOS/UEFI.
    • ~4 GB free RAM and ~5 GB disk for the image.
    • Three tools: KVM access, qemu, and nix (to build the image).

    That's it. The image is self-contained; everything else (the proxy, the in-guest firewall, the shell) is baked into it at build time.

    1. KVM¶

    Check that KVM is available and writable:

    ls -l /dev/kvm          # must exist
# optional sanity check:
sudo apt install cpu-checker && kvm-ok

    If /dev/kvm exists but you get a permission error later, add yourself to the kvm group and start a new login session:

    sudo usermod -aG kvm "$USER"
newgrp kvm              # or just log out and back in

    If /dev/kvm is missing, enable virtualization in your BIOS/UEFI.

    2. qemu¶

    # Debian / Ubuntu
sudo apt install qemu-system-x86 qemu-utils
# Fedora
sudo dnf install qemu-system-x86 qemu-img
# Arch
sudo pacman -S qemu-base

    3. nix (with flakes)¶

    The image is built with Nix flakes. The easiest installer is Determinate Systems' — it sets up the daemon and enables flakes by default:

    curl --proto '=https' --tlsv1.2 -sSf -L https://install.determinate.systems/nix | sh -s -- install

    (The upstream installer at https://nixos.org/download works too; the build script passes --extra-experimental-features 'nix-command flakes' so you do not need to edit nix.conf.)

    Open a new shell afterwards so nix is on your PATH.

    4. Get the code¶

    git clone <repo-url> katastrophe
cd katastrophe

    The build must run inside the git clone — it registers a couple of generated files with git add -fN so the flake can see them.

    You can drive everything two ways:

    • Directly via the bash scripts (no extra dependencies) — used below.
    • Via the kata CLI (nicer UX, optional). Build it with Rust/cargo: 
      cargo build --release --manifest-path cli/Cargo.toml
export PATH="$PWD/cli/target/release:$PATH"   # so `kata ...` works
      Then every bash scripts/<name>.sh below has a kata <name> equivalent (build-appliance → kata build-appliance, appliance → kata appliance, doctor → kata doctor).

    5. Check your host¶

    bash scripts/appliance.sh --help     # or: kata doctor

    kata doctor (if you built the CLI) reports KVM + qemu status. The appliance script also preflights these on every run.

    6. Build the appliance image¶

    bash scripts/build-appliance.sh      # or: kata build-appliance

    First build pulls a lot through Nix and takes a while. The result is a qcow2 under ~/.local/share/katastrophe/appliance-base.qcow2 (no sudo, no /var/lib).

    7. Run it¶

    mkdir -p /tmp/proj
bash scripts/appliance.sh --workspace /tmp/proj --ssh
    • --workspace DIR is shared into the guest at /workspace (read-write, over virtio-9p; host file ownership is preserved). It must be outside your home directory — a foot-gun guard refuses paths under $HOME. Use something like /tmp/proj or /srv/work. (To override for a trusted dir: FORBIDDEN_WORKSPACE_PREFIX= bash scripts/appliance.sh --workspace ~/p.)
    • --ro PATH shares an extra directory read-only at /mnt/ro_N (repeatable, up to 4). Unlike --workspace, --ro may live under your home — read-only ingress of a real project is the intended way to feed code in (qemu runs as you; the guest sees only that subtree).
    • --ssh boots the VM headless in the background and drops you straight into an SSH session (best for full-screen TUIs / truecolor). Drop it to use the serial console in the foreground instead.
    • Login user is agent, password agent. Poweroff from inside with sudo poweroff (or just exit the --ssh session).

    Run bash scripts/appliance.sh --help for all flags (--no-int for no network, memory/cpu sizing, warm bases via --from/--to, etc.).

    Networking & the allowlist¶

    By default the guest can only reach an allowlist of hostnames (Arch mirrors, PyPI, npm, GitHub, the Claude/Gemini/OpenAI endpoints, …). Everything else is refused by the in-guest proxy, and non-allowlisted egress never leaves the VM.

    To allow more hosts, edit the regex list in proxy/mitmproxy/allowlist.py and rebuild the image (step 6). Use --no-int for a fully offline session.

    Installing an agent CLI inside the guest¶

    The image ships the shell toolchain but not any agent CLI (kept generic). Inside the guest:

    npm i -g @anthropic-ai/claude-code     # for example
claude        # then /login

    To avoid re-installing every boot, bake it into a named warm base: --to mybase on a clean poweroff freezes the running session into a reusable base; --from mybase boots from it next time. See appliance.sh --help.

    Troubleshooting¶

    • /dev/kvm permission denied — you're not in the kvm group yet, or haven't started a fresh login session (newgrp kvm).
    • nix: command not found — open a new shell after installing nix, or source /nix/var/nix/profiles/default/etc/profile.d/nix-daemon.sh.
    • Path ... is not tracked during build — run the build from inside the git clone (not a copied/extracted tarball).
    • refusing --workspace under ... — your workspace is under $HOME; pick a dir elsewhere (see step 7).
    • Slow first boot — the appliance downloads agent-CLI deps on first use; freeze them into a warm base with --to so later boots are fast.

    Where things live¶

    • Image: ~/.local/share/katastrophe/appliance-base.qcow2
    • Warm bases: ~/.local/share/katastrophe/bases/
    • Per-session runtime state: $XDG_RUNTIME_DIR/katastrophe/

    Nothing is installed system-wide; remove the directories above and uninstall qemu/nix to fully clean up.
