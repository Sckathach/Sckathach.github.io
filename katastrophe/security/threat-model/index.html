    Threat model¶

    What this sandbox defends against, what it accepts, and where the load-bearing controls are. Component details are in Components; this page is the inventory of threats and their status.

    Reporting a vulnerability

    The reporting policy + supported-versions statement live in the repository's root SECURITY.md (so GitHub's "Report a vulnerability" UI can find it). This page is the threat inventory it points at.

    Legend: L likelihood · I impact · S status (accepted, mitigated, partial, open, deferred).

    The three profiles differ in where egress is enforced¶

    Profile Outer wall qemu/podman runs as A runtime escape lands as Egress gate
    Appliance KVM (strong) you you in-guest proxy + in-guest nft (defeatable by guest root, can't escape the VM)
    Hardened VM KVM (strong) agent (§5d drop) agent uid host nft (agent_isolate + agent_vm), survives guest compromise
    GPU user namespaces (weakest) agent uid agent uid host nft (agent_isolate + agent_uid_egress)

    Guest-root inside any VM is not an escape — it stays trapped behind KVM and touches nothing on the host. The "lands as" column is where a real hypervisor or namespace escape would put the attacker. Weak wall ↔ punishing host gate; strong wall ↔ the gate can be softer. The appliance trades the host gate away for zero host setup, so its allowlist is only as strong as the guest kernel.

    Every threat below maps to a runnable check on the Security invariants page — read this for the why, run those for the proof.

    Trust boundaries¶

    1. Agent uid (1001) — load-bearing for the GPU and hardened-VM paths. Kernel DAC, the agent_uid_egress skuid filter, the persistent agent_isolate skuid chain, and /dev/nvidia* ACL gating all hang on this uid. Anything crossing uid 1001 → another uid on the host (including root) breaches the sandbox. Since §5d, qemu in vm.sh also drops to this uid, so a hypervisor escape lands here too (instead of root).
    2. The host bridge — load-bearing for the hardened-VM path. The VM lives behind KVM; the bridge is the only host-exposed surface. The agent_vm table filters everything off the bridge.
    3. Host root / the host user. Out of scope — both can disable every control.
    4. The appliance has no host trust boundary beyond KVM itself: the proxy, the allowlist and the nft drop all live inside the disposable guest. This is the deliberate cost of needing no host setup.

    K.1 Kernel CVE reachable from userns (GPU path)¶

    • L/I/S: low / critical / accepted
    • kernel.unprivileged_userns_clone=1 is required for rootless podman. Historically vulnerable (CVE-2022-0185, CVE-2022-2602).
    • Mitigation: a hardened kernel + prompt kernel updates.

    K.2 NVIDIA driver CVE (GPU path)¶

    • L/I/S: low / critical / accepted
    • Proprietary module, large kernel surface, recurring priv-esc CVEs. Reachable from the agent uid via the session-scoped /dev/nvidia* ACL.
    • Mitigation: track driver CVEs and update promptly. The ACL is removed on session exit (kata disk umount / the GPU run's cleanup), but a compromise during a session lands in ring 0 instantly.

    K.3 sudo CVE¶

    • L/I/S: low / critical / accepted
    • vm.sh, net-up.sh and disk.sh run via sudo. A sudo bug lands host root.
    • Mitigation: keep sudo patched.

    K.4 Persistent agent_isolate chain is load-bearing¶

    • L/I/S: n/a / n/a / do not remove
    • Installed to /etc/nftables.d/ (or /etc/nftables.conf), agent_isolate drops skuid 1001 → RFC1918 + CGNAT (tailnet) + link-local + multicast. The GPU and hardened-VM paths rely on it for LAN / VPN / other-host isolation.
    • Removing it re-exposes private services to the agent. net-up.sh refuses to start if it isn't loaded. The appliance does not use it (no host nft) — its equivalent drop runs inside the guest.

    K.5 KVM / qemu escape (VM + appliance paths)¶

    • L/I/S: low / critical / accepted
    • KVM and qemu device emulation have had escapes (e.g. CVE-2019-14378). No PCIe passthrough in these builds — the emulated surface is virtio-net, virtio-blk, and vhost-user-fs (virtiofsd) / virtio-9p (appliance).
    • Mitigation: keep qemu patched; a hardened kernel reduces ring-0 surface. Since §5d the hardened VM runs qemu under -run-with user=agent, so an escape lands as the agent uid (caught by agent_isolate) rather than root. The appliance runs qemu as you, so an escape there lands as you — acceptable because that profile is for trying-it-out / no-API-key work, and a real escape past KVM already has your uid's reach.

    K.6 virtiofsd compromise (hardened VM path)¶

    • L/I/S: low / high / partial
    • virtiofsd runs as root on the host and serves the VM's /workspace (and any --rw/--ro mounts). A guest-side exploit against virtiofsd lands as host root — and unlike qemu (§5d), virtiofsd still needs root to traverse the shared trees, so it can't be dropped the same way.
    • Mitigation: --sandbox=chroot is set; upstream Rust virtiofsd has no published escapes to date. Limit --rw mounts to data the agent needs; prefer --ro ingress (the read-only path is the intended way to feed code in).
    • Open: no seccomp profile pinned for virtiofsd in this repo. (The appliance uses virtio-9p served by qemu-as-you, not root virtiofsd, so this threat does not apply there.)

    K.7 Bridge / nftables misconfig (hardened VM path)¶

    • L/I/S: low / high / mitigated
    • The VM reaches the internet only via the proxy. If agent_vm is missing or reordered, the VM could reach the host directly.
    • Mitigation: vm.sh preflights nft list table inet agent_vm and the bridge link, and refuses to start if either is missing. net-down.sh is idempotent — re-running net-up.sh resets the rules to a known state. kata verify reports the gate state.

    K.7b Proxy container hardening (mitmproxy / litellm)¶

    • L/I/S: low / med / mitigated
    • Both proxy containers run rootful podman, --network=host bound only to the RFC1918 proxy IP, --cap-drop=ALL, --security-opt=no-new-privileges, known upstream images, --read-only (litellm), under the default podman AppArmor profile.
    • Do not re-add apparmor=unconfined. An earlier revision unconfined both containers on a wrong diagnosis. The real startup failure was --cap-drop=ALL colliding with the mitmproxy image's root docker-entrypoint (a stat needing CAP_DAC_READ_SEARCH, then a su-exec needing CAP_SETUID/SETGID). The fix is --user 1000:1000 --entrypoint mitmdump, which bypasses the wrapper; both containers then run clean under the default profile.

    K.8 mitmproxy allowlist drift¶

    • L/I/S: med / med / accepted
    • Adding a host like gist.github.com opens a generic exfil channel. PyPI's publish endpoint, HF model-uploads, and GitHub gists are exfil-by-design.
    • Mitigation: keep the allowlist minimal; review proxy/mitmproxy/allowlist.py before appending.

    K.8b mitmproxy as a TLS-terminating CA (hardened path)¶

    • L/I/S: low / high / partial
    • The hardened mitmproxy holds a per-deployment CA private key (mode 0600) that the VM trusts. Anyone who can read that key can mint valid leaf certs for any allowlisted host as far as the VM is concerned.
    • Mitigation: per-deployment CA (not shared across hosts); the private key never leaves the host (only the public cert is published world-readable for the GPU container and baked into the VM trust store). The addon must never log request bodies. (The appliance does TLS passthrough, not termination, so it holds no CA — it can't read HTTPS bodies, only SNI/Host for allowlisting.)
    • Open: no automated CA rotation. Rotate by clearing the CA dir and re-running kata up (+ kata build-base to re-bake the VM trust store).

    K.8c DNS / raw-socket exfil (both host-gated paths)¶

    • L/I/S: low / high / mitigated
    • The proxy + probe only see HTTP(S) they intercept. Direct UDP/53 to a public resolver (DNS tunnelling) or raw/AF_PACKET frames would be channels they're blind to.
    • Mitigation: the agent needs no direct DNS — it reaches the proxy by IP. GPU: the terminal reject in agent_uid_egress drops all non-proxy egress including UDP/53. VM: agent_vm forward is policy drop, so the guest's direct egress is never routed. Verify from inside with scripts/audit-egress.sh (T1/T2).
    • Open: AF_PACKET reachability is argued safe (GPU uses pasta userspace networking; the VM bridge is NAT-routed, not L2-bridged to the wire) but not yet runtime-verified — audit-egress.sh T3 checks it; add a seccomp AF_PACKET block to gpu.sh if it turns out reachable.

    K.9 Storage persistence¶

    • L/I/S: low / med / accepted
    • The agent's work tree (@home) persists between sessions, so an implant survives. By design — real work needs persistence.
    • Mitigation: kata disk mount takes a read-only pre-session snapshot of @home/@state into the root:root @snapshots subvol (out of the agent's reach), kata disk snap takes an on-demand one before a risky run, and kata disk rollback restores one. Snapshots live on a subvolume the agent cannot enter, list, or btrfs subvolume delete.

    K.9b VM-base / snapshot poisoning (hardened path)¶

    • L/I/S: low / critical / mitigated
    • vm.sh parses base qcow2s, and qcow2 parsing is a code-execution surface. An agent-writable base or snapshot would be a host-code-exec path without a hypervisor escape.
    • Mitigation: @bases and @snapshots are you-/root-owned subvolumes with no agent write access — siblings of @home, never nested under it. The only writer of @bases is vm.sh --to, which runs as root. No agent-uid path writes a root-parsed base (verified, §5a).

    K.10 Compromised host user account¶

    • Out of scope. sudo access collapses the whole model.

    K.11 Physical access / firmware¶

    • Out of scope. Full-disk encryption (if you use the encrypted-USB storage layer) protects powered-off devices only.

    K.12 Side channels (Spectre / Meltdown / MDS / TAA)¶

    • Out of scope. Default kernel mitigations are assumed on; full side-channel isolation would need CPU pinning + SMT off, which this repo does not do.

    Notes¶

    • "Accepted" = an intentional trade-off given this threat model.
    • The paths are independent boundaries — escaping one doesn't bypass another. They share only the allowlist (K.8), the hardened CA (K.8b), and the kernel's CVE exposure (K.1, K.5).
