    Components & recipes¶

    katastrophe is a small kit of sandbox/egress components you can run and test on their own. The named profiles (appliance / hardened-vm / gpu) are just pre-wired recipes — points in a small space, not separate products.

    The components¶

    Component What it is Run it alone
    Isolator: VM throwaway qemu/KVM VM + workspace share scripts/appliance.sh
    Isolator: container rootless podman sandbox (+GPU via CDI) scripts/appliance-container.sh
    Egress proxy mitmproxy + hostname allowlist scripts/proxy.sh
    Probe probe(messages, response, meta) -> Verdict examples/probe-over-sample.py
    Netgate (hardened) host nft skuid drop + bridge + litellm scripts/net-up.sh (host integration, not a drop-in)

    The first four are genuinely standalone and rootless. The netgate is a host integration (it edits your nftables); read it for ideas rather than expecting to import it.

    What combines with what (the constraints)¶

    It's a constrained lattice, not a free grid:

    • GPU ⟹ container. Passing a single laptop GPU into a VM needs VFIO + a spare GPU; not practical. GPU work goes through the container isolator.
    • probe over bodies ⟹ terminating proxy. A passthrough proxy only sees the CONNECT host / SNI; it can't read request/response bodies.
    • terminating proxy ⟹ the client must trust the CA. Passthrough needs no CA anywhere (the appliance's whole point); terminating mode generates one and the isolator must trust it.
    • host-kernel egress ⟹ root + Linux + the nft plumbing, and pairs with the container (pasta carries the host uid → skuid-filterable) or a bridged VM. A rootless SLIRP VM can't be skuid-filtered, so it enforces egress in-guest.

    Egress enforcement strength¶

    Path Enforced where Survives a sandbox compromise?
    VM appliance in-guest proxy + nft no (guest root can disable), but can't escape the VM
    hardened (vm/gpu) host kernel (nft skuid drop) yes
    container appliance + proxy.sh proxy env (advisory) no — the workload can unset HTTP_PROXY

    So appliance-container.sh trades isolation strength for speed and zero setup. For a real egress boundary, use the VM appliance (in-guest) or the hardened path (host kernel).

    Recipes (the named profiles)¶

    Recipe = Command
    appliance VM + in-guest passthrough kata appliance --workspace DIR
    hardened vm VM + host netgate + terminating proxy + probe kata up then kata vm --workspace DIR
    gpu container + host netgate + terminating proxy + probe kata up then kata gpu --workspace DIR

    Demos¶

    All rootless, no host setup. Build the container image once: bash scripts/build-appliance-container.sh.

    Probe alone (no infra):

    python examples/probe-over-sample.py

    Proxy alone (point any client at it):

    bash scripts/proxy.sh                 # passthrough on 127.0.0.1:8080
# elsewhere:
HTTPS_PROXY=http://127.0.0.1:8080 curl https://pypi.org/      # allowed
HTTPS_PROXY=http://127.0.0.1:8080 curl https://example.com/   # 403, not allowed

    Container sandbox alone:

    mkdir -p /tmp/proj
bash scripts/appliance-container.sh --workspace /tmp/proj            # open egress
bash scripts/appliance-container.sh --workspace /tmp/proj --no-int   # offline

    Proxy in front of the container sandbox (the composition):

    bash examples/proxy-in-front.sh --workspace /tmp/proj            # passthrough
bash examples/proxy-in-front.sh --workspace /tmp/proj --probe    # terminate + probe (experimental)

    Inside the container:

    curl -sS -o /dev/null -w '%{http_code}\n' https://pypi.org/     # allowed
curl -sS https://example.com/                                   # denied

    Edit the allowlist in proxy/mitmproxy/allowlist.py.
