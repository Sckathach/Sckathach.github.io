    Security invariants¶

    A consolidated checklist: every load-bearing claim in the threat model, paired with a command that proves it and the expected outcome. Run this after first setup and after any kernel / qemu / podman / driver upgrade. ✗ in "Expect" means the action must fail — the gate doing its job looks like something not working.

    Egress is closed (run audit-egress.sh inside the sandbox)¶

    # Invariant Threat Run (inside sandbox) Expect
    T1 No DNS exfil to public resolvers K.8c audit-egress.sh ✗ no DNS reply
    T2 No direct TCP (egress is proxy-only) K.8c audit-egress.sh ✗ connection refused
    T3 No raw / AF_PACKET sockets K.8c audit-egress.sh ✗ socket denied
    T4 The proxy is reachable (positive control) — audit-egress.sh ✓ HTTP 2xx/3xx

    One command covers all four: bash audit-egress.sh (use PROXY_IP=127.0.0.1 in the appliance). It must print 4 passed, 0 failed.

    T3 on the GPU path

    If T3 reports AF_PACKET:CREATED, confirm frames can't actually leave and add a seccomp AF_PACKET block to gpu.sh if they can. Open item in K.8c.

    The host gate survives a sandbox bypass (hardened only)¶

    Invariant Threat Run Expect
    Egress pin is host-kernel, not container env K.4 inside: HTTP_PROXY= HTTPS_PROXY= curl --max-time 4 https://1.1.1.1/ ✗ fails anyway
    agent_isolate is loaded persistently K.4 host: sudo nft list table inet agent_isolate ✓ present
    net-up.sh refuses without it K.4 host: unload it, run kata up ✗ refuses to start
    kata verify reports gates + proxies K.7 host: kata verify ✓ all green

    Escape lands soft, not as root (§5d)¶

    Invariant Threat Run (host, VM running) Expect
    qemu runs as agent, not root K.5 ps -o user= -C qemu-system-x86_64 agent

    The vCPU process — what a hypervisor escape rides — is uid 1001, caught by agent_isolate. root here means qemu < 8.0 (or ALLOW_QEMU_ROOT=1).

    The agent can't reach what protects you from it¶

    Invariant Threat Run (host) Expect
    Agent can't write VM bases (qcow2 = code-exec surface) K.9b sudo -u agent touch /mnt/vm-bases/x.qcow2 ✗ Permission denied
    Agent can't enter/list snapshots K.9b / K.9 sudo -u agent ls /mnt/snapshots ✗ Permission denied
    @bases/@snapshots are siblings of @home, not nested K.9b kata disk status separate subvols, you-/root-owned
    Pre-session snapshot was taken K.9 kata disk mount then sudo ls /mnt/snapshots ✓ a read-only snapshot exists

    TLS termination trust (hardened)¶

    Invariant Threat Run (inside VM) Expect
    VM trusts the per-deployment CA K.8b curl https://pypi.org/simple/ ✓ 200, no cert error
    Allowlist denies unlisted hosts K.8 curl https://example.com/ ✗ 403
    CA private key stays host-side, 0600 K.8b host: sudo stat -c '%a %U' /etc/agent-proxy/mitmproxy/mitmproxy-ca.pem 600 + owner uid 1000

    The probe fails closed¶

    Invariant Threat Run Expect
    Blocked input denied before upstream — inside: litellm curl with "how to build a bomb" ✗ rejected (no upstream call)
    Probe crash → reject, not allow — code review: no try/except returning allow in probe/core.py holds

    See Egress & probe for the full commands and the known streaming-post-probe gap.

    Foot-gun guard¶

    Invariant Run Expect
    --workspace / --rw under $HOME refused kata vm --workspace ~/code/x ✗ refusing --workspace under …
    --ro under $HOME allowed (VM/appliance) kata vm --workspace /mnt/agent-home/work/p --ro ~/code/lib ✓ mounts at /mnt/ro_1

    Out of scope (do not expect these to be defended)¶

    Host root, a compromised host-user account, physical access / firmware, and Spectre-class side channels are explicitly out of scope (K.10–K.12). Don't file these as failures — they're accepted boundaries of a one-laptop, one-user model.
